SESSION 04

セキュリティと応用設計

60分 ｜ 座学 + 設計ワーク
今日のゴール

リスクを自分で判断できるようになり、「次にAI化するべき業務」を設計できるようになる

4-1

セキュリティ ― 5つのリスクを知る

15分

怖がるためではなく「何がどう危ないか」を知るためのパート。分かれば安全に使える。

レベル操作危険度メモ
1ファイルを読む★☆☆☆☆ほぼ安全
2ファイルを編集★★☆☆☆Gitで戻せる
3コマンド実行★★★☆☆確認画面を読む
4外部API連携（MCP）★★★★☆管理者が判断
5ブラウザ操作（MCP）★★★★★本当に必要か検討

普段の業務はレベル1〜3で十分。レベル4以上は必要になってから考えればいい。

5つのリスク

① ファイルの誤編集

Gitで「セーブ」しておけばいつでも戻せる。Gitが保険になる。

② 機密情報の送信

AIに見せたファイルはAnthropicに送られる。顧客名簿や個人情報を作業フォルダに置かない。

③ 危険なコマンドの実行

AIが npm install 〇〇 を実行しようとしたとき、その〇〇が安全か確認する。知らないパッケージ名は一度止めて調べる。

④ MCPで範囲が広がる

Google Sheets MCPを入れたら、そのシートの全データをAIが読める。必要最小限の権限で設定する。

⑤ コードの脆弱性

AIが書いたコードが完璧とは限らない。本番公開前に確認する。

4-2

実例で考える: D2Cブランドでの個人情報の扱い

10分

60社の会計データをClaude Codeで毎晩自動処理している税理士がいる。「会計データをAIに？」と感じるかもしれないが、実際のデータの流れはこうなっている。

全データがAIに送られているわけではない。8割はPC内のキーワード辞書で処理。Anthropicに送るのは判定できなかった2割だけ。しかもAPIで送ったデータはモデルの学習には使われない。

D2Cブランドの業務に置き換えると

やりたいこと扱うデータ注意点
メルマガ文面作成過去の配信テキスト個人情報なし。安全
DMチラシ制作商品情報、価格個人情報なし。安全
顧客分析購入履歴、名前、住所個人情報あり。匿名化が必要
レビュー分析ECサイトのレビュー匿名なら安全。名前付きは注意
覚えること

「AIで処理 ＝ 全データがAI会社に行く」ではない。データの流れを設計して送る範囲を最小限にできる。個人情報を扱う場合は匿名化してから渡す。

4-3

Git ― 保険をかける

10分 
セーブする = git commit（今の状態を記録）
戻す      = git checkout（過去の状態に戻る）
分岐する  = git branch（別パターンを試す）

ゲームのセーブと同じ。こまめにセーブしておけば、AIが何か壊しても戻せる。

GitHub = クラウド保管庫

自分のPCにしかないセーブデータをクラウドに上げて共有する。URLを伝えるだけで相手が取得できる。CLAUDE.mdもルールファイルも一緒に共有されるから、チーム全員が同じルールでAIを使える。

4-4

実践: 「次にAI化する業務」を設計する

20分

最終回のメインワーク。宿題で考えてきた「AI化したい業務」を実際に設計する。

■ 業務の概要
  何をやる業務か: ____________________
  現在の所要時間: 約____分/回
  頻度: ____回/月

■ 使っているツール
  □ スプレッドシート  □ メール  □ ECサイト管理画面
  □ Slack  □ Notion  □ その他: ________

■ 必要なMCP
  ツールにアクセスするために何が必要か:
  → 例: Google Sheets MCP、Gmail MCP

■ 必要な権限
  誰に何の許可をもらう必要があるか:
  → 例: EC管理画面のAPI権限は〇〇さんに申請

■ データの安全性
  □ 個人情報を含まない → そのまま使える
  □ 個人情報を含む → 匿名化の方法: ________

■ ルールファイルに書くこと
  この業務でAIに守らせたいルール:
  1. ____________________
  2. ____________________
  3. ____________________

■ まず試すスモールスタート
  全体をいきなりやるのではなく、最初に試す小さな範囲:
  → 例: まず1商品分だけ、まず1通分だけ

考え方の例

「毎週の在庫レポートをスプレッドシートからまとめてSlackに投稿している」なら:

→ Google Sheets MCPで在庫データを読み取る
→ ルールファイルにレポートのフォーマットを定義する
→ Slack MCPで投稿する
→ 権限: スプレッドシートの閲覧権限、SlackのBot権限
→ 個人情報: なし
→ スモールスタート: まず手動で1回やってみて、出力を確認
ゴールのイメージ

この設計ができるようになれば、新しい業務課題が出てきたときに「〇〇とMCP接続して、〇〇の権限をもらえれば自動化できる。あとは要件定義をどう詰めるか」と判断できる。そこまでいければ、AIエージェントの導入は自走できる。

4-5

社内ルール確認

5分 
ルール1: 専用の作業フォルダで作業する
ルール2: APIキーをコードに書かない
ルール3: 顧客の個人情報はAIに読ませない（匿名化してから）
ルール4: AIの実行許可は内容を確認してから
ルール5: MCPの追加は管理者に相談
ルール6: ルールファイルを勝手に消さない

この6つだけ覚えておく。困ったらこのリストに戻る。

4-6

全4回のまとめ

回学んだこと
第1回AIエージェントは「自分で手を動かすAI」。便利だが範囲管理が必要。
第2回ターミナルは文字版Finder。Claude CodeはPC上で動き、頭脳はAnthropicにある。作業フォルダが最初の安全策。
第3回ルールファイルでブランドのトンマナをAIに教えられる。MCPで外部ツールと接続できるが、1つずつ管理する。
第4回5つのリスクと対策。個人情報の扱い方。Gitは保険。「何をAI化するか」を自分で設計する力。

AIエージェントは「優秀だけど事情を知らない外注さん」。ブランドマニュアル（ルールファイル）を渡して、作業範囲（フォルダ）を決めて、成果物を確認して、安全に共有する（GitHub）。この流れさえ押さえれば、どんな業務にも応用できる。

卒業課題

自分の担当業務を1つ、AIエージェントで改善してみる

手順:
  ① 第4回で作った設計シートをもとに作業フォルダを準備する
  ② CLAUDE.md にブランドルールと業務ルールを書く
  ③ 業務に必要なデータを作業フォルダに入れる
    （個人情報は除外 or 匿名化）
  ④ AIに指示を出して、業務の一部を実行させる
  ⑤ 出力を確認し、ルールを修正・追記する
  ⑥ 改善前と改善後の比較をまとめる

提出するもの

  • CLAUDE.md の中身
  • AIの出力結果（スクショまたはファイル）
  • 「うまくいったこと」「うまくいかなかったこと」のメモ
  • 次にやりたいこと（あれば）
