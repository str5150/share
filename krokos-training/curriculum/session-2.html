SESSION 02

PCの仕組みを理解する

60分 ｜ 座学 + ハンズオン
今日のゴール

ターミナルが何をする場所か、AIエージェントが自分のPC上で何をしているかを理解する

2-1

ターミナルとは

12分

普段はマウスでフォルダを開いたりファイルを削除したりしている。ターミナルは同じことを「文字」でやる場所。

【Finderでやっていること】            【ターミナルで同じこと】
フォルダをダブルクリックして開く  →  cd my-project
フォルダの中身を見る              →  ls
右クリック→「新規フォルダ」      →  mkdir new-folder
ファイルをゴミ箱にドラッグ        →  rm file.txt

やっていることは全く同じ。違いは「マウスで操作するか、文字で命令するか」だけ。

ハンズオン: ターミナルとFinderを並べて確認する

ターミナルとFinderを横に並べて、両方で同じ場所を見ながらやるとわかりやすい。

1. ターミナルを開く（Macなら「ターミナル」アプリ）

2. pwd と打つ
   → 「/Users/自分の名前」と表示される
   → これはFinderの「ホーム」フォルダと同じ場所

3. ls と打つ
   → Desktop, Documents, Downloads... と表示される
   → Finderのホームフォルダを開くと、同じフォルダが並んでいる

4. cd Desktop と打つ
   → デスクトップに移動した
   （Finderでデスクトップフォルダを開いたのと同じ）

5. mkdir ai-test と打つ
   → Finderのデスクトップを見ると
   「ai-test」フォルダが出現している

6. ls と打つ
   → ai-test が一覧に入っている

ターミナルで打ったコマンドの結果が、そのままFinderに反映されている。逆もそうで、Finderでフォルダを作ればターミナルの ls にも出る。同じPCの同じファイルを、別の窓から操作しているだけ。

ポイント

AIエージェントは、このターミナルを使って作業する。mkdir でフォルダを作り、ファイルを書き込み、コマンドを実行する。つまりAIがターミナルを使うということは、AIがFinderでできることを全部自動でやれるということ。だから「何をしていいか」を管理する必要がある。

ターミナルでできること（Finder以上のことも）

ファイルのダウンロード（curl）    ← ブラウザを開かなくてもDLできる
ソフトのインストール（brew install）← App Storeを開かなくてもインストール
サーバーの起動（npx serve）       ← 作ったページをブラウザで確認
GitHubへのアップロード（git push） ← ファイルをクラウドに送信
2-2

「実行する（Run）」と「止める」

3分 
実行 = プログラムを動かすこと
  node app.js  ← これで app.js が動く

止める = Ctrl + C
  サーバーを起動したとき、画面が止まって見えることがある。
  壊れたのではなく、動き続けているだけ。Ctrl + C で止められる。

「実行したけど画面が固まった」はよくあること。フリーズではなく、プログラムが動き続けている状態。

2-3

Homebrew・PATH・bin

5分

深入りは不要。「こういう仕組みがある」と知っておくだけでいい。

Homebrew = ターミナル版のApp Store
  → brew install node でツールをインストールできる

bin = ツール置き場
  → インストールしたツールは /usr/local/bin/ などに置かれる

PATH = ツールの探し方リスト
  → ターミナルでコマンドを打つと、PATHに書かれた場所を順番に探す
  → 見つからないと「command not found」エラーになる
トラブルシューティング

「command not found」が出たらPCが壊れたわけではない。ツールが置いてある場所をPCが知らないだけ。トラブル時に「PATHの問題かも」と思えるだけで対応が変わる。

2-4

AIエージェントはどこで何をしているのか

10分

3つのポイント

  1. AIが触るファイルは自分のPCの中にある
  2. AIの頭脳はAI提供元のサーバーにある（CodexならOpenAI、Claude CodeならAnthropic）
  3. AIに見せた情報はインターネット経由で送信される

作業フォルダの話

AIエージェントは起動したフォルダの中で動く。「AIに触らせていいファイルだけが入ったフォルダ」で起動する。これが一番基本のセキュリティ。

2-5

AIエージェントの安全設定 ― 7つの基本

15分

AIエージェントが自分のPCで動くことがわかった。次に必要なのは「どこまで許すか」の設定。

実際にあった事故

AIエージェント（Claude Code）が git push --force を実行しかけた事例がある。原因は差分確認のダイアログで「Always allow（常に許可）」を確認せずに連打していたこと。かなり重要なリポジトリだったため、冷や汗ものだったという。

公式ドキュメントにもこう書いてある:

Claude Code only has the permissions you grant it.
You're responsible for reviewing proposed code
and commands for safety before approval.

つまり「セキュリティは自分で設定しろ」ということ。使う人が自分で安全ラインを引く必要がある。

基本の4つ ― 最低限やっておくこと

① サンドボックス（隔離環境）を有効にする

AIが実行するコマンドをOSレベルで隔離する機能。macOSではSeatbelt、LinuxではBubble Wrapが使われる。

Claude Codeの場合:
{
  "sandbox": {
    "enabled": true,
    "allowUnsandboxedCommands": false   ← これが重要
  }
}

有効にしただけでは不十分。allowUnsandboxedCommands: false を設定して初めて脱出口が完全に塞がる。「有効化」と「完全に塞ぐ」は別物。

② 危険なコマンドをブロックする（denyルール）

「絶対に実行させたくないコマンド」を事前に登録しておく。denyルールは最優先で、後から「Always allow」にしても上書きされない。

ブロック推奨コマンド:
  rm -rf *           ← 全削除
  git push --force * ← 強制プッシュ（取り返しがつかない）
  git reset --hard   ← 強制リセット（取り返しがつかない）
  curl / wget        ← 外部通信（デフォルトでもブロックだが明示推奨）
  chmod 777          ← 権限全開放
③ 機密ファイルへのアクセスを塞ぐ

.env ファイル（APIキーが入っている）、SSH鍵、AWSクレデンシャルなど、AIに読ませたくないファイルを明示的にブロックする。AIのツール経由と、ターミナルコマンド経由の両方を設定するのが確実。

Claude Codeの場合:
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(**/*.pem)",
      "Read(**/*.key)"
    ]
  },
  "sandbox": {
    "filesystem": {
      "denyRead": ["~/.aws/credentials", "~/.ssh"]
    }
  }
}

プロンプトインジェクション攻撃（悪意あるコードにAIへの指示が埋め込まれるケース）で、AIが意図せず機密ファイルを読もうとする可能性がある。読み取り自体をブロックしておくことが重要。

④ ネットワークのアクセス先を制限する

AIがアクセスできるドメインをホワイトリスト方式で制限する。GitHub、npm、PyPIなど業務に必要なドメインだけ許可し、それ以外はブロック。悪意あるコードがAIを操って外部サーバーにデータを送信しようとしても、ホワイトリスト外への通信はブロックされる。

応用の3つ ― 知っておくと強い

設定概要
⑤ 実行前フックコマンド実行前にカスタムスクリプトで安全チェック。denyルールで対応しきれない複雑な条件に使える
⑥ 権限の棚卸し「Always allow」で蓄積した許可ルールを月1回チェック。不要な許可を削除する
⑦ 組織ポリシーの強制管理者が全員に共通の安全設定を配信。個人が勝手にルールを変更できなくする。MCPサーバーも管理者が許可したものだけに限定可能

現実的な使い分け

全部を常時オンにする必要はない。作業内容に応じて切り替える。

作業内容推奨レベル
LPや資料の作成①②で十分
APIキーを使う開発①②③
外部サービスと連携①②③④
顧客データに触れる作業①〜⑥すべて
チーム全体での運用①〜⑦すべて

「何も設定していない状態」と「基本だけでも設定した状態」では安全性がまったく違う。

2-6

APIとは

5分 
チャットAI = レストランで食事
  → ブラウザという店に行って、その場で質問して回答をもらう

AIエージェント = 出前
  → 自分のPC（キッチン）からAPI（電話）で回答（料理）を届けてもらう
  → 届いた結果を自分のPCで好きに使える

APIキーは出前の会員番号。漏れたら他人に使われて課金が発生する。パスワードと同じ扱い。

2-7

振り返り

10分
Q1: ターミナルの ls コマンドは何をする？
ファイルの一覧を表示する。FinderやExplorerで中身を見るのと同じ。
Q2: AIエージェントを起動するとき「作業フォルダ」が重要な理由は？
AIが触れるファイルの範囲が決まるから。顧客データが入ったフォルダで起動すると危険。
Q3: 「command not found」エラーの意味は？
PATHに登録された場所にプログラムがない。PCが壊れたわけではない。
Q4: AIエージェントの安全設定で最低限やるべきことは？
サンドボックスの有効化、denyルールで危険コマンドをブロック、機密ファイルへのアクセス制限、ネットワーク制限の4つ。

宿題

ターミナルで以下のコマンドを打ってみる

pwd           ← 今いる場所を表示
ls            ← ファイル一覧
cd Desktop    ← デスクトップに移動
mkdir ai-work ← 「ai-work」フォルダを作る
cd ai-work    ← そのフォルダに入る
pwd           ← 場所が変わったことを確認

フォルダの安全性を書き出す

自分のPCのフォルダ構成を見て、「AIに触らせても安全なフォルダ」と「絶対ダメなフォルダ」を1つずつ書き出す。
