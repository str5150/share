# Claude Code 導入講座 資料パック

> 対象: チャットAI（ChatGPT等）は使えるが、GitHubやターミナルは触ったことがない方
> 形式: 1回60分 × 全4回
> ゴール: AIエージェントを使って自社ブランドの業務を自走で改善できるようになる

---

## 1. 講座概要

### この講座で身につくこと

- チャットAIとAIエージェントの違いを説明できる
- ターミナルと作業フォルダの意味を理解して安全に使える
- ルールファイルを使ってブランドのトンマナをAIに反映できる
- MCPの役割とリスクを理解し、必要な権限を整理できる
- 次にAI化する業務を自分で設計できる

### 受講前の前提

- ChatGPTなどのチャットAIを触ったことがある
- GitHubやターミナルは未経験でも問題ない
- ブランド業務、販促、EC、運用業務のいずれかに関わっている

### 全体の流れ

```text
第1回  AIエージェントを知る
       チャットAIとの違いを体感し、最初の指示を出す

第2回  PCの仕組みを理解する
       ターミナル、フォルダ、APIの基礎を押さえる

第3回  ブランド業務に適用する
       ルールファイルとMCPで実務ワークに取り組む

第4回  セキュリティと応用設計
       リスク管理を学び、次に何をAI化するか設計する
```

## 2. 第1回: AIエージェントを知る

> 今日のゴール: チャットAIとAIエージェントの違いを体感し、実際に1つ指示を出してみる

### アジェンダ

| セクション | 時間 | ねらい |
| --- | --- | --- |
| 1-1. 生成AIの整理 | 10分 | AIの種類を整理して不安を減らす |
| 1-2. チャットAI vs AIエージェント | 15分 | 「答えるAI」と「手を動かすAI」の差を理解する |
| 1-3. AIエージェントツールの種類 | 5分 | Claude Code、Cursorなどの位置づけを知る |
| 1-4. 実践: 最初の指示を出す | 20分 | 実際にAIがファイルを作る体験をする |
| 1-5. 振り返り | 10分 | 学びを言語化し、宿題につなげる |

### 1-1. 生成AIの整理

| AI | 開発元 | ひとことで言うと |
| --- | --- | --- |
| GPT | OpenAI | 万能型。何でもそこそこやる |
| Claude | Anthropic | 慎重で丁寧。長文に強い |
| Gemini | Google | 検索との連携が強い |
| Llama | Meta | オープンソース。自社サーバーで動かせる |

講師メモ:

- 覚えるべき本質は「どのAIを使うか」より「どう使うか」
- この講座ではClaude Codeを使うが、考え方は他のエージェントにも応用できる

### 1-2. チャットAI vs AIエージェント

**チャットAI**

```text
あなた → 質問する → AIが文章で回答 → あなたがコピペして作業する
```

**AIエージェント**

```text
あなた → 「商品紹介LPを作って」
      → AIがファイルを作る
      → エラーが出たら修正する
      → 完成したら報告する
```

### RPAとの比較

| 項目 | RPA | AIエージェント |
| --- | --- | --- |
| たとえると | レールの上の電車 | 自分で道を決めるタクシー |
| 想定外のこと | 止まる | 自分で考える |
| 指示の出し方 | フロー図を組む | 日本語で伝える |

講師メモ:

- RPAは安心感、AIエージェントは柔軟性
- ただしAIエージェントは範囲管理がないと危ない

### 1-3. AIエージェントツールの種類

| ツール | ひとこと |
| --- | --- |
| Claude Code | ターミナルで動く。速い。腕利きの職人 |
| Cursor | 画面付きエディタ。変更が目に見える。隣に座る助手 |
| Devin / Antigravity | クラウド上で完結。リモートの外注チーム |

補足:

- 見た目は違っても、裏でやっていることは「ファイルの読み書き + コマンド実行」

### 1-4. 実践: 最初の指示を出してみる

**やること**

- デスクトップに `test-page` フォルダを作る
- Claude CodeまたはCursorをそのフォルダで起動する
- 商品紹介ページを1枚作らせる

**指示例**

```text
シンプルな商品紹介ページを1枚作って。
商品名は〇〇。
特徴を3つ並べて、購入ボタンを置いて。
```

**観察ポイント**

- AIがファイルを作成している
- エラーがあれば修正している
- 許可確認のタイミングがある

### 1-5. 振り返り

- Q1. チャットAIとAIエージェントの一番大きな違いは何か
- Q2. RPAとAIエージェント、想定外の対応に強いのはどちらか
- Q3. Claude Code、Cursor、Devinで共通していることは何か

**宿題**

- 今日作ったページをスクショして保存する
- 自分の業務で毎回やっている定型作業を3つ書き出す

## 3. 第2回: PCの仕組みを理解する

> 今日のゴール: ターミナルが何をする場所か、Claude Codeが自分のPC上で何をしているかを理解する

### アジェンダ

| セクション | 時間 | ねらい |
| --- | --- | --- |
| 2-1. ターミナルとは | 15分 | 文字でPCを操作する感覚を身につける |
| 2-2. 実行する・止める | 5分 | フリーズとの違いを理解する |
| 2-3. Homebrew・PATH・bin | 10分 | よくあるエラーの見方を知る |
| 2-4. Claude Codeはどこで何をしているか | 15分 | ローカルとクラウドの役割を理解する |
| 2-5. APIとは | 5分 | APIキーの意味をつかむ |
| 2-6. 振り返り | 10分 | 安全に使う前提を整理する |

### 2-1. ターミナルとは

普段の操作を文字でやる場所、と伝える。

| 普段のやり方 | ターミナルで同じこと |
| --- | --- |
| フォルダを開く | `cd my-project` |
| ファイル一覧を見る | `ls` |
| フォルダを作る | `mkdir new-folder` |
| ファイルを削除する | `rm file.txt` |

**ハンズオン**

```text
pwd
ls
mkdir test-folder
ls
```

講師メモ:

- Finderの文字版、という理解で十分
- 難しい概念に見せず、結果が目に見えることを重視する

### 2-2. 実行する・止める

```text
実行 = プログラムを動かす
止める = Ctrl + C
```

ポイント:

- 画面が止まって見えてもフリーズとは限らない
- サーバーは動き続けるものだと体感してもらう

### 2-3. Homebrew・PATH・bin

| 用語 | ひとことで言うと |
| --- | --- |
| Homebrew | ターミナル版のApp Store |
| bin | ツール置き場 |
| PATH | ツールの探し方リスト |

補足:

- `command not found` は「壊れた」ではなく「見つからない」

### 2-4. Claude Codeはどこで何をしているのか

```text
あなたのPC
  ├ Claude Code
  └ 作業フォルダ（ファイル読み書き）

インターネット経由

Anthropic社のサーバー
  └ ここで考えている
```

**3つのポイント**

1. AIが触るファイルは自分のPCにある
2. AIの頭脳はAnthropicのサーバーにある
3. AIに見せた情報はインターネット経由で送られる

### 2-5. APIとは

たとえ話:

- チャットAI = レストランで食事
- Claude Code = 出前

APIキーは会員番号や鍵に近い。漏らすと他人に使われる。

### 2-6. 振り返り

- Q1. `ls` は何をするコマンドか
- Q2. 作業フォルダが重要な理由は何か
- Q3. `command not found` の意味は何か

**宿題**

```text
pwd
ls
cd Desktop
mkdir ai-work
cd ai-work
pwd
```

- AIに触らせても安全なフォルダ
- 絶対に触らせたくないフォルダ

この2つを自分のPC上で書き出してくる。

## 4. 第3回: ブランド業務に適用する

> 今日のゴール: ルールファイルとMCPを使って、自社ブランドのトンマナや業務データをAIに活かす方法を身につける

### アジェンダ

| セクション | 時間 | ねらい |
| --- | --- | --- |
| 3-1. AIエージェントの内部動作 | 10分 | 指示から完了までの流れを理解する |
| 3-2. ルールファイル | 15分 | ブランドルールの渡し方を学ぶ |
| 3-3. 実践① メルマガのトンマナをルール化 | 15分 | 実務データをルールへ落とし込む |
| 3-4. MCP | 10分 | 外部サービス連携の考え方を学ぶ |
| 3-5. 実践② DMチラシ素材づくり | 10分 | ルールを使った成果物生成を体験する |

### 3-1. AIエージェントの内部動作

```text
STEP 1 指示を出す
STEP 2 AIが計画を立てる
STEP 3 フォルダの中身を確認する
STEP 4 ファイルを作成・編集する
STEP 5 動作確認・修正する
STEP 6 完了報告する
```

講師メモ:

- 2から5が何度もループしている
- チャットAIは1往復、エージェントは反復作業を自動で行う

### 3-2. ルールファイル

構成例:

```text
my-project/
├ index.html
├ style.css
└ CLAUDE.md
```

**CLAUDE.md の例**

```md
# ブランドルール
- ブランド名は「〇〇」表記で統一
- メインカラーは #1a73e8、サブカラーは #f5f5f5
- フォントは Noto Sans JP を使う
- トーンは「親しみやすく、でも安っぽくない」
- 価格は税込表記
- 「お客様」ではなく「あなた」で呼びかける
```

ポイント:

- 学習ではなくマニュアル
- 誰でも読める
- チームで共有できる

### 3-3. 実践①: メルマガのトンマナをルール化する

**手順**

```text
1. 作業フォルダを作る
2. 過去メルマガを2〜3本入れる
3. AIに文体・トンマナ・NGワードを分析させる
4. CLAUDE.md を生成させる
5. 内容を見て修正する
6. ルールありで新規メルマガを書かせる
```

確認ポイント:

- ルールなしだと汎用的になる
- ルールありだとブランドらしさが出る

### 3-4. MCP

素のClaude Codeでできること:

- ファイルを読む
- ファイルを書く
- コマンドを実行する

MCP追加で増えること:

- Google Sheets MCP
- Web Search MCP
- Slack MCP

講師メモ:

- MCPは「道具箱を1つずつ足す」感覚
- 1個入れたら何でもできるわけではない

### 3-5. 実践②: DMチラシの素材を作る

**手順**

```text
1. product-info.txt を作る
2. 商品名、特徴、価格、ターゲットを書く
3. AIにキャッチコピー3案と本文を依頼する
4. CLAUDE.md のルール順守を確認する
5. 修正指示を出す
```

**修正指示の例**

```text
2案目をもう少しカジュアルにして
価格表記を税込に変えて
```

**宿題**

- CLAUDE.md に足りない項目を3つ以上追加する
- 次回までにAI化したい業務を1つ決める

## 5. 第4回: セキュリティと応用設計

> 今日のゴール: リスクを自分で判断できるようになり、「次にAI化するべき業務」を設計できるようになる

### アジェンダ

| セクション | 時間 | ねらい |
| --- | --- | --- |
| 4-1. セキュリティ | 15分 | 5つのリスクを理解する |
| 4-2. 実例で考える | 10分 | データの流れを具体的に把握する |
| 4-3. Git | 10分 | セーブポイントという認識を持つ |
| 4-4. 実践: 次にAI化する業務を設計 | 20分 | 業務改善を自分で設計する |
| 4-5. 社内ルール確認 | 5分 | 最低限の運用ルールを定める |

### 4-1. セキュリティ

**危険度レベル**

| レベル | 内容 | 危険度 | 基本方針 |
| --- | --- | --- | --- |
| 1 | ファイルを読む | 低 | 基本的に安全 |
| 2 | ファイルを編集する | 中低 | Gitで戻せる |
| 3 | コマンド実行 | 中 | 確認画面を読む |
| 4 | 外部API連携 | 高 | 管理者判断 |
| 5 | ブラウザ操作 | 高 | 本当に必要か検討 |

**5つのリスク**

1. ファイルの誤編集
2. 機密情報の送信
3. 危険なコマンドの実行
4. MCPで範囲が広がる
5. コードの脆弱性

### 4-2. 実例で考える

ブランド業務への当てはめ:

| やりたいこと | 扱うデータ | 注意点 |
| --- | --- | --- |
| メルマガ文面作成 | 過去の配信テキスト | 個人情報なしなら安全 |
| DMチラシ制作 | 商品情報、価格 | 個人情報なしなら安全 |
| 顧客分析 | 購入履歴、名前、住所 | 匿名化が必要 |
| レビュー分析 | ECレビュー | 名前付きなら注意 |

伝えるべきこと:

- AI処理 = 全データ送信、ではない
- 送る範囲は設計できる
- APIとチャットでは扱いが違う

### 4-3. Git

```text
セーブする = git commit
戻す      = git checkout
分岐する  = git branch
```

GitHubはクラウド保管庫。チーム共有と復元の両方に効く。

### 4-4. 実践: 「次にAI化する業務」を設計する

**設計シート**

```text
■ 業務の概要
何をやる業務か:
現在の所要時間:
頻度:

■ 使っているツール
スプレッドシート / メール / EC管理画面 / Slack / Notion / その他

■ 必要なMCP

■ 必要な権限

■ データの安全性
個人情報を含むか:
匿名化の方法:

■ ルールファイルに書くこと
1.
2.
3.

■ まず試すスモールスタート
```

例:

```text
毎週の在庫レポートをスプレッドシートからまとめてSlackに投稿する
```

ここから:

- Google Sheets MCPが必要
- Slack MCPが必要
- 閲覧権限とBot権限が必要
- 個人情報なし
- まず1回だけ手動で試す

### 4-5. 社内ルール確認

```text
1. 専用の作業フォルダで作業する
2. APIキーをコードに書かない
3. 顧客の個人情報はAIに読ませない
4. AIの実行許可は内容を確認してから
5. MCPの追加は管理者に相談する
6. ルールファイルを勝手に消さない
```

## 6. 全4回のまとめ

- 第1回: AIエージェントは自分で手を動かすAI
- 第2回: 作業フォルダとターミナルが安全運用の土台
- 第3回: ルールファイルでブランドらしさを渡せる
- 第4回: リスクを理解するとAI化の設計ができる

最後に伝えるメッセージ:

> AIエージェントは「優秀だけど事情を知らない外注さん」。
> ブランドマニュアルを渡し、作業範囲を決め、成果物を確認し、安全に共有する。
> この流れを押さえれば、どんな業務にも応用できる。

## 7. 卒業課題

```text
課題: 自分の担当業務を1つ、AIエージェントで改善してみる
```

**手順**

1. 設計シートをもとに作業フォルダを準備する
2. `CLAUDE.md` にブランドルールと業務ルールを書く
3. 必要なデータを入れる
4. AIに指示を出す
5. 出力を確認し、ルールを修正する
6. 改善前後を比較する

**提出物**

- `CLAUDE.md` の内容
- AIの出力結果
- うまくいったこと
- うまくいかなかったこと
- 次にやりたいこと
